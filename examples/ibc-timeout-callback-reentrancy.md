## Patterns

- `OnTimeoutPacket` calling mint or burn operations
- `OnTimeoutPacket` modifying state before forwarding to next middleware
- Middleware implementing `IBCModule` with token operations in timeout handler
- `OnRecvPacket` and `OnTimeoutPacket` sharing mutable state without guards

## Assessment

IBC middleware that mints/burns tokens in timeout callbacks can be exploited via reentrancy if a malicious relayer triggers timeout and receive for the same packet in specific sequences. The timeout handler may execute token operations (mint refund tokens) while the packet is still being processed by another middleware in the stack, leading to infinite mint.

Check:
1. Does `OnTimeoutPacket` perform token mint/burn/transfer operations?
2. Is there shared mutable state between `OnRecvPacket` and `OnTimeoutPacket`?
3. Are there reentrancy guards (mutex, processing flag) on the packet lifecycle?
4. Can a relayer trigger timeout for a packet that has already been partially processed?

If (1) is yes and (3) is no → likely vulnerable.

Reference: ASA-2024-007 (ibc-go), infinite token mint via CosmWasm middleware reentrancy.
